```
  → No aprovecha la volatilidad del mercado
  → Las ganancias micro no se generan
```

## 3. Descripción de los 3 Bugs

### Bug 1 — Order Splitter destruye las órdenes (el más crítico)

**Causa raíz:**

El Order Splitter divide cada orden en 3 micro-órdenes (30/40/30%).
Con capital pequeño, cada micro-orden cae por debajo del mínimo
notional de Binance ($5 USDC), y Binance las rechaza silenciosamente.

```
Ejemplo con $40 USDC activo / 6 niveles = $6.67 por nivel:

  Con SPLIT_ENABLED=true (comportamiento actual):
    Micro-orden 1: $6.67 × 30% = $2.00 USDC ← RECHAZADA (< $5)
    Micro-orden 2: $6.67 × 40% = $2.67 USDC ← RECHAZADA (< $5)
    Micro-orden 3: $6.67 × 30% = $2.00 USDC ← RECHAZADA (< $5)
    Resultado: 0 de 3 micro-órdenes colocadas por nivel

  Con SPLIT_ENABLED=false (fix):
    Orden única: $6.67 USDC ← ACEPTADA (> $5)
    Resultado: 1 orden por nivel, 6 órdenes totales ✅
```

### Bug 2 — Re-análisis sobreescribe la config manual

**Causa raíz:**

El análisis automático de mercado (Capa 3 o config-selector)
corre periódicamente y recomienda una configuración diferente
a la que el usuario definió manualmente en el .env.
El bot acepta esa recomendación y reconstruye el grid con
parámetros distintos (generalmente más conservadores = más separados).

```
Flujo problemático:
  Usuario define: GRID_RANGE_PERCENT=1, GRID_LEVELS=6
  Bot arranca con esa config ✅
  
  15 minutos después → Análisis automático corre
  Volatilidad baja → recomienda config CONSERVATIVE
  Config CONSERVATIVE: GRID_RANGE_PERCENT=6, GRID_LEVELS=8
  
  Bot reconstruye grid con config CONSERVATIVE ❌
  Las órdenes ahora están muy separadas
  El usuario no ve sus parámetros reflejados
```

### Bug 3 — Threshold de reconstrucción demasiado bajo

**Causa raíz:**

El bot reconstruye el grid cuando el precio se acerca al límite
del rango. Si el threshold es muy bajo (ej. 50%), el bot
reconstruye constantemente con pequeños movimientos de precio,
no permitiendo que el grid capture la volatilidad normal.

```
Ejemplo con threshold al 50%:
  Grid: $1.346 - $1.374 (rango de $0.028)
  Trigger de reconstrucción: cuando precio toca el 50% del rango
  = cuando el precio se mueve $0.007 desde el centro
  = cada vez que XRP se mueve 0.5% ← demasiado frecuente
  
  XRP se mueve 0.5% cada pocos minutos en días normales
  → El grid se reconstruye constantemente
  → Nunca captura un ciclo completo de compra + venta
```

## 4. Fix 1 — Desactivar Order Splitter con capital pequeño

### 4.1 Cambio en .env y .env.example

```env
# Cambiar:
SPLIT_ENABLED=false

# Agregar comentario explicativo en .env.example:
# Order Splitter divide cada orden en 3 micro-órdenes (30/40/30%).
# Con capital pequeño esto hace que las micro-órdenes caigan
# por debajo del mínimo notional de Binance ($5 USDC) y sean rechazadas.
# Desactivar cuando: capital_activo / GRID_LEVELS < $15 USDC
# Activar cuando:    capital_activo / GRID_LEVELS > $15 USDC
SPLIT_ENABLED=false
```

### 4.2 Agregar validación automática en el arranque

En `lib/bot/scheduler.ts` o donde se inicializa el grid,
agregar una validación que detecte automáticamente si el
splitter causará problemas y lo desactive:

```typescript
// Después de calcular el capital activo y los niveles:

const capitalPerLevel = activeCapitalUSDC / config.GRID_LEVELS
const minNotional = 5  // mínimo de Binance en USDC
const splitParts = 3   // 30/40/30

// Si la micro-orden más pequeña (30%) cae bajo el mínimo:
const smallestMicroOrder = capitalPerLevel * 0.30

if (config.SPLIT_ENABLED && smallestMicroOrder < minNotional) {
  logger.warn(
    `⚠️  Order Splitter desactivado automáticamente. ` +
    `Capital por nivel: $${capitalPerLevel.toFixed(2)} USDC. ` +
    `Micro-orden mínima: $${smallestMicroOrder.toFixed(2)} USDC ` +
    `< mínimo Binance $${minNotional} USDC. ` +
    `Para activar el splitter necesitas al menos ` +
    `$${(minNotional / 0.30 * config.GRID_LEVELS).toFixed(0)} USDC activo total.`
  )
  // Override temporal en memoria (no modifica el .env)
  runtimeConfig.splitEnabled = false
} else {
  runtimeConfig.splitEnabled = config.SPLIT_ENABLED
}
```

### 4.3 Mostrar en dashboard

### 5.3 Cambio en layer3-agent.ts

La respuesta del agente Claude puede incluir `grid_adjustment`
con action `rebuild` o `narrow`. Con MANUAL_GRID_CONFIG=true,
el bot debe ignorar los ajustes de grid pero sí respetar
las alertas de riesgo:

```typescript
// En lib/analysis/layer3-agent.ts
// Después de parsear la respuesta del agente:

function applyAgentResponse(response: Layer3AgentResponse): void {

  // Alertas de riesgo: SIEMPRE aplicar
  if (response.risk_flags.length > 0) {
    handleRiskFlags(response.risk_flags)
  }

  // Ajustes de grid: solo si no está en modo manual
  if (config.MANUAL_GRID_CONFIG) {
    if (response.grid_adjustment.action !== 'keep') {
      logger.info(
        `Agente recomendó: ${response.grid_adjustment.action} ` +
        `pero MANUAL_GRID_CONFIG=true — ignorando ajuste de grid. ` +
        `Razón del agente: ${response.grid_adjustment.reason}`
      )
    }
    // No aplicar el ajuste de grid
    return
  }

  // Aplicar ajuste de grid normalmente
  applyGridAdjustment(response.grid_adjustment)
}
```

## 6. Fix 3 — Aumentar threshold de reconstrucción

### 6.1 Nueva variable en .env

```env
# Porcentaje del rango que debe alcanzar el precio
# para triggear una reconstrucción del grid.
# 
# 50% = reconstruye cuando el precio toca la mitad del rango
#       desde el extremo → muy frecuente, inestable
# 80% = reconstruye cuando el precio está al 80% del límite
#       → más estable, captura más ciclos completos
# 90% = casi hasta el límite → máxima estabilidad
#
# Para sesiones de 1 hora con volatilidad normal de XRP:
# Recomendado: 85
GRID_REBUILD_THRESHOLD=85
```

### 6.2 Cambio en risk.ts o engine.ts

Buscar donde se evalúa si el precio está cerca del límite
del grid y triggea la reconstrucción. Actualizar la lógica:

```typescript
// Buscar algo similar a esto en el código actual:
// if (currentPrice <= gridMin || currentPrice >= gridMax)

// Reemplazar con threshold configurable:

function shouldRebuildGrid(
  currentPrice: number,
  gridMin: number,
  gridMax: number
): { shouldRebuild: boolean; reason: string } {

  const gridRange = gridMax - gridMin
  const gridCenter = (gridMax + gridMin) / 2
  const threshold = config.GRID_REBUILD_THRESHOLD / 100  // ej: 0.85

  // Distancia máxima permitida desde el centro
  // antes de reconstruir
  const maxDistance = (gridRange / 2) * threshold

  const distanceFromCenter = Math.abs(currentPrice - gridCenter)
  const distancePercent = (distanceFromCenter / (gridRange / 2)) * 100

  if (distancePercent >= config.GRID_REBUILD_THRESHOLD) {
    return {
      shouldRebuild: true,
      reason: `Precio al ${distancePercent.toFixed(1)}% del límite ` +
              `(threshold: ${config.GRID_REBUILD_THRESHOLD}%)`
    }
  }

  // También reconstruir si el precio salió completamente del rango
  if (currentPrice <= gridMin || currentPrice >= gridMax) {
    return {
      shouldRebuild: true,
      reason: `Precio fuera del rango: $${currentPrice} ` +
              `(rango: $${gridMin} - $${gridMax})`
    }
  }

  return { shouldRebuild: false, reason: '' }
}
```

### 6.3 Log informativo del threshold

Cada ciclo, si el precio se acerca al threshold, loguear
una advertencia antes de que se reconstruya:

```typescript
const { shouldRebuild, reason } = shouldRebuildGrid(
  currentPrice, gridMin, gridMax
)

// Advertencia preventiva cuando está al 70% del threshold
const warningThreshold = config.GRID_REBUILD_THRESHOLD * 0.85
if (distancePercent >= warningThreshold && !shouldRebuild) {
  logger.warn(
    `⚠️  Precio acercándose al límite: ${distancePercent.toFixed(1)}% ` +
    `(reconstrucción a: ${config.GRID_REBUILD_THRESHOLD}%)`
  )
}

if (shouldRebuild) {
  logger.info(`🔄 Reconstruyendo grid: ${reason}`)
  await rebuildGrid()
}
```
