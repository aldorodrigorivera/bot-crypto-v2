## 1. Contexto — Por qué existe v3

El bot v2 tiene una brecha crítica identificada al compararlo con sistemas
de trading algorítmico profesionales: **opera sin validación histórica previa**.

Cualquier configuración de grid (niveles, rango, par) se activa directamente
sin saber si esa misma configuración habría sido rentable en los últimos meses.
Esto es operar con fe, no con evidencia.

El framework profesional correcto es:
```
Research → Backtest → Incubate → Operate
```

El bot v1 y v2 solo hacen la última parte. El v3 agrega las tres anteriores
como pasos obligatorios antes de que el grid se active con dinero real.

Adicionalmente, v3 agrega:
- Métricas de evaluación formales (Win Rate, Profit Factor, Sharpe, Drawdown)
- Modo de incubación (operar con tamaño mínimo antes de escalar)
- Multi-estrategia: el bot puede correr y comparar configuraciones en paralelo
- Un motor de backtesting completo sobre datos históricos reales de Binance

### 6.2 simulator.ts — Simulador del Grid

Este es el corazón del backtesting. Debe replicar EXACTAMENTE la lógica
del grid real tal como está implementada en `src/bot/grid.ts`.

```typescript
interface SimulationConfig {
  candles: OHLCVCandle[]
  gridConfig: {
    levels: number
    rangePercent: number
    baseAmount: number      // SIZING_BASE_AMOUNT del .env
  }
  feePercent: number        // 0.1 para Binance spot
  pair: string
}

interface SimulationResult {
  trades: SimulatedTrade[]
  finalCapital: number
  startCapital: number
  totalReturn: number        // en %
  durationDays: number
}

interface SimulatedTrade {
  type: 'buy' | 'sell'
  price: number
  amount: number
  timestamp: number
  fee: number
  profit?: number            // solo en sells: ganancia del ciclo
  cycleId: string            // une la compra con su venta correspondiente
}
```

#### Algoritmo del simulador paso a paso

```
INICIALIZACIÓN:
  1. Tomar precio de apertura de la primera vela como precio inicial
  2. Construir grid: misma fórmula que grid.ts en producción
     → calcular precio de cada nivel igual que lo hace el bot real
  3. Capital inicial = baseAmount × levels (USDC equivalente)
  4. Registrar órdenes abiertas: todas las de compra debajo del precio,
     todas las de venta arriba del precio

POR CADA VELA (high y low en ese orden):
  1. Verificar si el HIGH de la vela toca alguna orden de VENTA abierta
     → Si sí: ejecutar la venta, calcular ganancia, registrar trade
     → Colocar nueva orden de COMPRA un nivel abajo
  
  2. Verificar si el LOW de la vela toca alguna orden de COMPRA abierta
     → Si sí: ejecutar la compra, registrar trade
     → Colocar nueva orden de VENTA un nivel arriba

  3. Verificar si el precio salió del rango del grid:
     → Si close < gridMin o close > gridMax: registrar evento "grid roto"
     → El simulador reconstruye el grid centrado en el nuevo precio
       (igual que lo haría el bot real al detectar ruptura)

FEES:
  Aplicar 0.1% sobre el valor de cada orden ejecutada (compra Y venta)
  La ganancia neta por ciclo = (precio_venta - precio_compra) × amount
                               - fee_compra - fee_venta

RESULTADO FINAL:
  Retornar todos los trades simulados para que metrics.ts los procese
```

**Regla crítica de fidelidad:** El simulador debe usar la MISMA lógica de
construcción del grid que `src/bot/grid.ts`. Si grid.ts cambia en el futuro,
el simulador debe actualizarse también. Considerar extraer la lógica de
construcción del grid a una función pura compartida por ambos.

### 6.3 metrics.ts — Métricas Formales

```typescript
interface BacktestMetrics {
  // Métricas principales
  winRate: number              // % de ciclos (compra+venta) con ganancia > 0
  profitFactor: number         // suma_ganancias / suma_pérdidas (en valor absoluto)
  sharpeRatio: number          // retorno_promedio / desviación_estándar_retornos
  maxDrawdown: number          // % de caída máxima del capital desde un pico

  // Métricas secundarias
  totalTrades: number          // total de operaciones (compras + ventas)
  completedCycles: number      // pares compra+venta completados
  totalReturn: number          // % de retorno total sobre el capital inicial
  avgProfitPerCycle: number    // ganancia promedio por ciclo completo en USDC
  avgDuration: number          // duración promedio de un ciclo en minutos
  totalFeesPaid: number        // fees totales pagados en USDC
  netProfitUSDC: number        // ganancia neta total en USDC
  gridBreaks: number           // cuántas veces el precio rompió el rango

  // Evaluación
  passed: boolean              // ¿cumple todos los benchmarks del .env?
  failedReasons: string[]      // qué benchmarks no cumplió (si passed=false)
  score: number                // score compuesto 0-100 para comparar configs
}
```

#### Cálculo de cada métrica

**Win Rate:**
```
cyclesWithProfit = ciclos donde (precio_venta - precio_compra - fees) > 0
winRate = (cyclesWithProfit / totalCycles) × 100
```

**Profit Factor:**
```
grossProfit = suma de todas las ganancias positivas de ciclos
grossLoss   = suma del valor absoluto de todas las pérdidas de ciclos
profitFactor = grossProfit / grossLoss
Si grossLoss = 0: profitFactor = Infinity (todos los ciclos ganaron)
```

**Sharpe Ratio (simplificado, sin tasa libre de riesgo):**
```
returns = array de % de retorno por cada ciclo completado
avgReturn = promedio(returns)
stdReturn = desviación_estándar(returns)
sharpe = avgReturn / stdReturn × sqrt(ciclos_por_año_estimados)
```

**Max Drawdown:**
```
Recorrer el equity curve (capital acumulado en cada punto del tiempo)
peak = máximo histórico hasta ese punto
drawdown_en_t = (peak - capital_en_t) / peak × 100
maxDrawdown = máximo de todos los drawdown_en_t
```

**Score compuesto (para comparar configuraciones entre sí):**
```
score = (winRate × 0.30)
      + (min(profitFactor, 3) / 3 × 100 × 0.30)
      + (min(sharpeRatio, 3) / 3 × 100 × 0.20)
      + ((20 - min(maxDrawdown, 20)) / 20 × 100 × 0.20)

Resultado entre 0 y 100.
Una config con win rate 60%, PF 1.8, Sharpe 1.2, DD 8% → score ~72
```

**Evaluación de benchmarks (passed/failedReasons):**
```
Tomar valores de .env:
  BACKTEST_MIN_WIN_RATE     → winRate >= este valor
  BACKTEST_MIN_PROFIT_FACTOR → profitFactor >= este valor
  BACKTEST_MAX_DRAWDOWN     → maxDrawdown <= este valor
  BACKTEST_MIN_SHARPE       → sharpeRatio >= este valor
  BACKTEST_MIN_TRADES       → totalTrades >= este valor

Si TODOS se cumplen → passed = true
Si alguno falla → passed = false, agregar a failedReasons
```

### 6.4 runner.ts — Multi-Config Runner

Solo se activa si `MULTI_CONFIG_ENABLED=true`.

```typescript
// Las 3 configs definidas en v2 (conservative, balanced, aggressive)
// Se importan desde config/index.ts donde ya están definidas

async function runMultiConfigBacktest(
  candles: OHLCVCandle[]
): Promise<MultiConfigResult>

interface MultiConfigResult {
  conservative: BacktestMetrics
  balanced: BacktestMetrics
  aggressive: BacktestMetrics
  winner: 'conservative' | 'balanced' | 'aggressive'
  winnerReason: string    // "Mayor score compuesto (72/100). Win rate 61%, PF 1.8"
}
```

**Comportamiento:**
1. Correr `simulator.ts` tres veces, una por cada config, sobre los MISMOS datos
2. Calcular métricas para cada una con `metrics.ts`
3. Comparar scores compuestos
4. La config con mayor score → `winner`
5. Si la ganadora no pasa los benchmarks mínimos → ninguna pasa, bot no arranca

### 7.1 Modificación a scheduler.ts

**después del análisis de mercado y antes de la confirmación del usuario:**

  4. Análisis de mercado
  5. Capa 3 — análisis inicial del agente

  1. Express + WS levantan
  2. Back4App inicializa
  3. ¿Estado previo? → reanudar o nuevo
  4. Análisis de mercado
  5. [NUEVO] Si MULTI_CONFIG_ENABLED: runner.ts → elegir mejor config
  6. [NUEVO] Si !MULTI_CONFIG_ENABLED: usar config del análisis de mercado
  7. [NUEVO] BACKTEST de la config elegida (si BACKTEST_ENABLED=true)
             → Mostrar resultados del backtest en consola
             → ¿Pasa benchmarks? → continuar
             → ¿No pasa? → mostrar qué falló → preguntar al usuario
  8. Capa 3 — análisis inicial del agente (recibe métricas del backtest)
  9. Mostrar configuración + métricas del backtest → confirmación usuario
  10. [NUEVO] Si INCUBATION_ENABLED: iniciar en modo incubación
      Si no: iniciar grid con tamaños normales

## 8. Módulo de Incubación

### 8.1 manager.ts — Gestión del modo incubación

El modo incubación es el puente entre el backtest y la operación real a
tamaño completo. Es obligatorio si `INCUBATION_ENABLED=true`.

```typescript
interface IncubationState {
  isActive: boolean
  startedAt: Date
  currentPhase: IncubationPhase
  realTrades: number           // trades reales ejecutados hasta ahora
  realWinRate: number          // win rate real (no simulado)
  realProfitFactor: number
  currentSizeMultiplier: number // qué fracción del tamaño normal se usa
  totalRealProfitBTC: number
  totalRealLossBTC: number
  passedAt?: Date              // cuándo salió de incubación (si ya salió)
}

type IncubationPhase =
  | 'micro'      // tamaño mínimo (INCUBATION_MIN_SIZE)
  | 'small'      // 25% del tamaño normal
  | 'medium'     // 50% del tamaño normal
  | 'normal'     // 100% — salió de incubación

```

### 8.2 scaler.ts — Escalado gradual

```
REGLAS DE ESCALADO:

Fase MICRO → SMALL:
  Requisito: >= 10 trades reales Y win rate real >= 50%
  Acción: sizeMultiplier pasa de mínimo a 0.25
  Mensaje: "Incubación: escalando a tamaño SMALL (25%)"

Fase SMALL → MEDIUM:
  Requisito: >= 20 trades reales Y win rate real >= 53%
             Y sin pérdida acumulada mayor a INCUBATION_MAX_LOSS_PERCENT
  Acción: sizeMultiplier pasa a 0.50
  Mensaje: "Incubación: escalando a tamaño MEDIUM (50%)"

Fase MEDIUM → NORMAL:
  Requisito: >= INCUBATION_MIN_TRADES trades reales
             Y >= INCUBATION_DURATION_DAYS días en incubación
             Y win rate real >= INCUBATION_TARGET_WIN_RATE
             Y sin pérdida acumulada mayor a INCUBATION_MAX_LOSS_PERCENT
  Acción: sizeMultiplier = 1.0 → incubación completada
  Mensaje: "✅ Incubación completada — Operando a tamaño NORMAL"

ABORT de incubación:
  Si pérdida acumulada > INCUBATION_MAX_LOSS_PERCENT:
  → Detener bot
  → Guardar estado en Back4App con stopReason = 'incubation_loss_limit'
  → Mensaje: "❌ Incubación abortada — Pérdida límite alcanzada"
  → El backtest necesita revisión antes de reintentar
```

### 8.3 monitor.ts — Monitoreo durante incubación

Cada ciclo principal, además del monitoreo normal del bot, verificar:

1. ¿Se cumplieron los criterios para escalar de fase?
2. ¿Se superó el límite de pérdida?
3. Emitir evento WebSocket `incubation_update` con el estado actual
4. Guardar snapshot en `IncubationState` en Back4App cada 10 trades

## 14. Notas Finales para Claude Code

1. **El backtesting es el corazón de v3.** Si hay tiempo limitado, priorizar
   Fases 1, 2 y 3 sobre todo lo demás. Un backtest que funcione correctamente
   en consola ya es una mejora enorme aunque el dashboard no tenga las
   secciones nuevas todavía.

2. **El simulador debe ser fiel al bot real.** La tentación es simplificar
   la simulación. No hacerlo. Si el bot real reconstruye el grid cuando el
   precio rompe el rango, el simulador debe hacer lo mismo. De lo contrario
   el backtest sobreestima el rendimiento.

3. **Los datos históricos cuestan requests a Binance.** El cache de 30 minutos
   en `dataLoader.ts` es importante para no hacer miles de requests innecesarios
   durante el desarrollo y las pruebas.

4. **La incubación no reemplaza al position sizer de v2.** La relación es:
   `tamañoFinal = positionSizer(capas1y2, agente) × incubationMultiplier`
   Si incubación no está activa, `incubationMultiplier = 1.0` (sin efecto).

5. **Los benchmarks del .env son configurables por diseño.** Un usuario con
   un perfil de riesgo alto puede bajar `BACKTEST_MIN_WIN_RATE` a 50%.
   Un usuario conservador puede subirlo a 65%. El código no debe hardcodear
   estos valores — siempre leerlos del config.

6. **Si el backtest no tiene suficientes trades** (`totalTrades < BACKTEST_MIN_TRADES`),
   el veredicto debe ser `passed = false` con razón:
   `"Muestra insuficiente: solo X trades simulados, mínimo requerido: Y.
    Considera aumentar BACKTEST_DAYS o reducir BACKTEST_MIN_TRADES."`

7. **El modo incubación persiste entre reinicios.** Si el bot se apaga
   mientras está en incubación y se reinicia, debe retomar la incubación
   desde donde estaba (leyendo `IncubationState` de Back4App), no empezar
   desde cero.

8. **La sección Real vs Simulado del dashboard** solo aparece cuando hay
   suficientes trades reales (≥ 20). Antes de eso mostrar un placeholder:
   `"Esperando suficientes trades reales para la comparación... (X/20)"`
