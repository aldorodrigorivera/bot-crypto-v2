## Contexto — Por qué existe v5

El grid actual se construye simétricamente alrededor del precio:
igual número de niveles arriba que abajo, igual tamaño en todos.

El problema: el precio no se mueve simétricamente. Hay presión
compradora o vendedora en cada momento. Un grid simétrico ignora
esa información y deja dinero sobre la mesa.

La idea central de v5:

```
ANTES (v1-v4):
  "¿Dónde está el precio ahora?"
  → Grid centrado ahí, simétrico

NUEVO (v5):
  "¿Hacia dónde fluirá el precio en las próximas horas?"
  → Grid asimétrico, sesgado hacia donde hay más probabilidad
  → Más niveles donde el precio probablemente irá
  → Menos niveles donde probablemente no irá
```

Esto no es predicción de precio. Es gestión de probabilidades
basada en evidencia observable en tiempo real.

## Las 4 señales del LiquidityAnalyzer

### Señal 1 — Order Book Imbalance (OBI)

**Qué mide:** Ratio entre presión compradora y vendedora en el
order book en tiempo real.

**Fuente de datos:** `exchange.fetchOrderBook('XRP/USDC', 20)`
— top 20 niveles de bids y asks.

```typescript
interface OBIResult {
  bidVolume: number        // USDC total en los top 10 bids
  askVolume: number        // USDC total en los top 10 asks
  ratio: number            // bidVolume / askVolume
  bias: 'bullish' | 'bearish' | 'neutral'
  strength: number         // 0-100
}

// Cálculo:
// ratio > 1.5  → bullish fuerte  (strength 70-100)
// ratio > 1.2  → bullish moderado (strength 40-70)
// ratio 0.8-1.2 → neutral        (strength 0-40)
// ratio < 0.8  → bearish moderado
// ratio < 0.5  → bearish fuerte

// Ejemplo:
// bids: $180,000 USDC / asks: $90,000 USDC
// ratio = 2.0 → bullish fuerte, strength 85
```

**Cómo afecta el grid:**
```
OBI bullish fuerte  → más niveles de VENTA (arriba del precio)
OBI bearish fuerte  → más niveles de COMPRA (abajo del precio)
OBI neutral         → grid simétrico (comportamiento v4)
```

---

### Señal 2 — Cumulative Volume Delta (CVD)

**Qué mide:** Si el volumen neto de las últimas N horas fue de
compras agresivas (market buys) o ventas agresivas (market sells).

**Fuente de datos:** `exchange.fetchTrades('XRP/USDC', 500)`
— últimas 500 transacciones del mercado.

```typescript
interface CVDResult {
  buyVolume: number        // volumen de market buys
  sellVolume: number       // volumen de market sells
  delta: number            // buyVolume - sellVolume
  cumulativeDelta: number  // delta acumulado últimas 4 horas
  trend: 'accumulation' | 'distribution' | 'neutral'
  strength: number         // 0-100
}

// Cómo clasificar:
// delta positivo + creciendo   → accumulation (dinero entrando)
// delta negativo + cayendo     → distribution (dinero saliendo)
// delta oscilando cerca de 0  → neutral

// Ejemplo:
// market buys: 450,000 XRP
// market sells: 210,000 XRP
// delta: +240,000 XRP → accumulation fuerte
```

**Cómo afecta el grid:**
```
CVD accumulation → confirma sesgo alcista del OBI
CVD distribution → confirma sesgo bajista del OBI
CVD contradice OBI → reducir fuerza del sesgo (señales mixtas)
```

### Señal 3 — Liquidity Map (Mapa de Clusters)

**Qué mide:** Dónde están los niveles de precio con mayor
concentración histórica de órdenes — soportes y resistencias
basados en liquidez real, no en líneas dibujadas.

**Fuente de datos:**
`exchange.fetchOrderBook('XRP/USDC', 100)` — order book profundo
+ historial de OHLCV últimas 48 horas para identificar niveles
donde el precio "rebotó" repetidamente.

```typescript
interface LiquidityLevel {
  price: number
  volume: number           // volumen acumulado en ese nivel
  type: 'support' | 'resistance' | 'neutral'
  strength: number         // 0-100 (cuántas veces actuó como soporte/resistencia)
  distanceFromCurrent: number  // % de distancia del precio actual
}

interface LiquidityMap {
  levels: LiquidityLevel[]
  nearestResistance: LiquidityLevel  // cluster más cercano arriba
  nearestSupport: LiquidityLevel     // cluster más cercano abajo
  currentZone: 'near_resistance' | 'near_support' | 'in_range'
}
```

**Cómo afecta el grid:**

```
El grid se construye HACIA los clusters de liquidez:

Ejemplo:
  Precio actual: $1.40
  Cluster resistencia más cercano: $1.44 (strength 80)
  Cluster soporte más cercano: $1.37 (strength 60)

  Sin v5: grid de $1.34 a $1.46 (simétrico ±4%)
  
  Con v5: concentrar más niveles entre $1.40 y $1.44
          porque ahí es donde habrá más actividad
          
  Resultado: 8 niveles entre $1.40-$1.44 (denso)
             4 niveles entre $1.37-$1.40 (escaso)
```

### Señal 4 — Funding Rate

**Qué mide:** Si el mercado de futuros de XRP está sesgado
hacia posiciones largas o cortas. Indica el "sentimiento"
del dinero institucional.

**Fuente de datos:** `exchange.fetchFundingRate('XRP/USDC')`

```typescript
interface FundingRateResult {
  rate: number             // tasa actual (ej: 0.0003 = 0.03%)
  interpretation: 'longs_paying' | 'shorts_paying' | 'neutral'
  riskLevel: 'low' | 'medium' | 'high'
  note: string             // explicación en lenguaje natural
}

// Interpretación:
// rate > +0.05%  → longs pagando mucho → mercado muy largo
//                  → riesgo de long squeeze → cuidado sesgo alcista
// rate > +0.01%  → longs pagando → sesgo alcista moderado ✅
// rate ~0%       → neutral → no afecta el sesgo
// rate < -0.01%  → shorts pagando → sesgo bajista moderado
// rate < -0.05%  → shorts pagando mucho → riesgo de short squeeze
```

**Cómo afecta el grid:**
```
Funding muy positivo (> +0.05%):
  OVERRIDE: reducir sesgo alcista aunque OBI y CVD sean bullish
  El mercado ya está "muy largo" y puede corregir

Funding muy negativo (< -0.05%):
  OVERRIDE: reducir sesgo bajista aunque señales sean bearish
  El mercado ya está "muy corto" y puede rebotar

Funding neutral (±0.01%):
  No modifica el sesgo — OBI y CVD dominan la decisión
```

## El GridBias — Output del LiquidityAnalyzer

Todas las señales se combinan en un único objeto `GridBias`
que el grid engine usa para construir el grid asimétrico:

```typescript
interface GridBias {
  // Dirección general
  direction: 'bullish' | 'bearish' | 'neutral'
  strength: number              // 0-100 (fuerza del sesgo)
  confidence: number            // 0-100 (cuántas señales coinciden)

  // Distribución de niveles
  levelsAbove: number           // cuántos niveles poner arriba
  levelsBelow: number           // cuántos niveles poner abajo
  // levelsAbove + levelsBelow = GRID_LEVELS total del config

  // Densidad del grid
  densityZone: {
    priceMin: number            // inicio de la zona densa
    priceMax: number            // fin de la zona densa
    levelConcentration: number  // % de niveles a concentrar aquí
  }

  // Tamaño por dirección
  sizeMultiplierAbove: number   // multiplicador para órdenes arriba
  sizeMultiplierBelow: number   // multiplicador para órdenes abajo

  // Señales individuales (para logging y dashboard)
  signals: {
    obi: OBIResult
    cvd: CVDResult
    liquidityMap: LiquidityMap
    fundingRate: FundingRateResult
  }

  // Override activo
  overrideActive: boolean
  overrideReason?: string       // ej: "Funding rate muy alto, reduciendo sesgo"

  // Para la Capa 3
  summary: string               // resumen en lenguaje natural para Claude
}
```

## Algoritmo de combinación de señales

```typescript
function calculateGridBias(
  obi: OBIResult,
  cvd: CVDResult,
  liquidityMap: LiquidityMap,
  fundingRate: FundingRateResult,
  totalLevels: number           // GRID_LEVELS del config activo
): GridBias {

  // 1. Score base por señal (cada una aporta entre -100 y +100)
  //    Positivo = alcista, Negativo = bajista
  const obiScore = obi.bias === 'bullish' ? obi.strength
                 : obi.bias === 'bearish' ? -obi.strength : 0

  const cvdScore = cvd.trend === 'accumulation' ? cvd.strength
                 : cvd.trend === 'distribution' ? -cvd.strength : 0

  const liquidityScore = liquidityMap.currentZone === 'near_support' ? 30
                       : liquidityMap.currentZone === 'near_resistance' ? -20 : 0

  // 2. Score ponderado
  //    OBI tiene más peso porque es la señal más inmediata
  const weightedScore = (obiScore * 0.40)
                      + (cvdScore * 0.35)
                      + (liquidityScore * 0.25)

  // 3. Confidence: cuántas señales apuntan en la misma dirección
  const signalsAligned = [obiScore, cvdScore, liquidityScore]
    .filter(s => Math.sign(s) === Math.sign(weightedScore)).length
  const confidence = (signalsAligned / 3) * 100

  // 4. Override por funding rate extremo
  let finalScore = weightedScore
  let overrideActive = false
  let overrideReason = ''

  if (fundingRate.rate > 0.05 && weightedScore > 0) {
    finalScore = weightedScore * 0.3   // reducir fuertemente el sesgo alcista
    overrideActive = true
    overrideReason = 'Funding rate muy positivo — riesgo de long squeeze'
  } else if (fundingRate.rate < -0.05 && weightedScore < 0) {
    finalScore = weightedScore * 0.3   // reducir fuertemente el sesgo bajista
    overrideActive = true
    overrideReason = 'Funding rate muy negativo — riesgo de short squeeze'
  }

  // 5. Calcular distribución de niveles
  //    Score +100 → 80% de niveles arriba, 20% abajo
  //    Score 0    → 50% arriba, 50% abajo (simétrico)
  //    Score -100 → 20% arriba, 80% abajo
  const biasRatio = 0.5 + (finalScore / 100) * 0.3
  // biasRatio va de 0.2 a 0.8

  const levelsAbove = Math.round(totalLevels * biasRatio)
  const levelsBelow = totalLevels - levelsAbove

  // 6. Multiplicadores de tamaño por dirección
  //    El lado favorecido recibe órdenes más grandes
  const sizeMultiplierAbove = finalScore > 0
    ? 1.0 + (finalScore / 100) * 0.3   // hasta 1.3x en el lado alcista
    : 1.0 - (Math.abs(finalScore) / 100) * 0.2  // hasta 0.8x si bajista

  const sizeMultiplierBelow = finalScore < 0
    ? 1.0 + (Math.abs(finalScore) / 100) * 0.3
    : 1.0 - (finalScore / 100) * 0.2

  return {
    direction: finalScore > 20 ? 'bullish'
             : finalScore < -20 ? 'bearish' : 'neutral',
    strength: Math.abs(finalScore),
    confidence,
    levelsAbove,
    levelsBelow,
    densityZone: {
      priceMin: finalScore > 0
        ? liquidityMap.levels[0]?.price ?? 0   // zona entre precio actual y resistencia
        : liquidityMap.nearestSupport?.price ?? 0,
      priceMax: finalScore > 0
        ? liquidityMap.nearestResistance?.price ?? 0
        : liquidityMap.levels[0]?.price ?? 0,
      levelConcentration: 60   // 60% de los niveles en la zona densa
    },
    sizeMultiplierAbove,
    sizeMultiplierBelow,
    signals: { obi, cvd, liquidityMap, fundingRate },
    overrideActive,
    overrideReason,
    summary: buildSummary(finalScore, confidence, obi, cvd, fundingRate, overrideActive)
  }
}
```

## Modificación a grid.ts — Grid Asimétrico

El grid engine actual construye niveles equidistantes alrededor
del precio. Se modifica para aceptar un `GridBias` opcional:

```typescript
// ANTES (firma actual):
function buildGrid(
  currentPrice: number,
  config: GridConfig
): GridLevel[]

// DESPUÉS (nueva firma):
function buildGrid(
  currentPrice: number,
  config: GridConfig,
  bias?: GridBias          // opcional — si undefined: grid simétrico v4
): GridLevel[]
```

**Comportamiento con bias:**

```
Sin bias (undefined):
  Comportamiento idéntico a v4
  Grid simétrico, niveles equidistantes

Con bias neutral (strength < 20):
  Comportamiento idéntico a v4
  No hay suficiente evidencia para sesgar

Con bias bullish (strength ≥ 20):
  levelsAbove: bias.levelsAbove (ej: 8)
  levelsBelow: bias.levelsBelow (ej: 4)
  Niveles arriba más densos en la densityZone
  Tamaño de órdenes arriba × sizeMultiplierAbove

Con bias bearish (strength ≥ 20):
  levelsAbove: bias.levelsAbove (ej: 4)
  levelsBelow: bias.levelsBelow (ej: 8)
  Niveles abajo más densos en la densityZone
  Tamaño de órdenes abajo × sizeMultiplierBelow
```

```typescript
// 3. El precio toca un cluster de liquidez fuerte
```

## Consideraciones importantes para Claude Code

1. **El bias es opcional en grid.ts.** Si `bias` es `undefined` o
   `bias.direction === 'neutral'` con `strength < 20`, el grid
   se construye idéntico a v4. Nunca romper el comportamiento anterior.

2. **Los datos de mercado son siempre reales.** El `MOCK_BALANCE`
   solo mockea el balance y las órdenes, no el order book ni los
   trades del mercado. El LiquidityAnalyzer siempre usa datos reales.

3. **El CVD requiere muchos trades.** `fetchTrades` retorna las
   últimas N transacciones pero puede no cubrir 4 horas completas
   en momentos de bajo volumen. Implementar fallback: si hay menos
   de 100 trades disponibles, usar timeframe más corto y marcarlo
   en el resultado (`lowDataWarning: true`).

4. **El funding rate puede no estar disponible.** XRP/USDC es un
   par spot. El funding rate es de futuros (XRP/USDT perp en Binance).
   Usar `fetchFundingRate('XRP/USDT')` en lugar de XRP/USDC.
   Si falla → ignorar esta señal (funding = neutral).

5. **El LiquidityMap no es un indicador técnico clásico.** No usar
   `technicalindicators` para esto. Construirlo desde el order book
   profundo y el historial de OHLCV directamente.

6. **Loguear todo el proceso.** Cada vez que se calcula un GridBias,
   el log debe mostrar las 4 señales individuales y cómo se combinaron.
   Esto es crítico para debuggear si el grid se comporta inesperadamente.

## Expectativa de mejora realista

```
Con grid simétrico (v4):
  El precio sube → ejecutas ventas y capturas ganancia ✅
  El precio baja → ejecutas compras y capturas ganancia ✅
  Pero: igual capital en ambas direcciones
  
Con grid asimétrico (v5, sesgo bullish 73/100):
  El precio sube → 8 niveles de venta con más capital ✅✅
  El precio baja → 4 niveles de compra con menos capital ✅
  Pero: si el sesgo estaba equivocado y el precio baja fuerte,
  tienes menos protección hacia abajo

Mejora esperada cuando el sesgo acierta: +20-40% más ganancia
Riesgo cuando el sesgo falla: -10-20% menos protección

Por eso la confianza (confidence) importa:
  Confidence 100% (3/3 señales alineadas) → aplicar bias completo
  Confidence 67% (2/3 señales alineadas) → aplicar bias al 70%
  Confidence 33% (1/3 señales alineadas) → aplicar bias al 30%
  Confidence 0% (0/3 señales alineadas)  → grid simétrico
```

### El bot tiene dos usos distintos de IA

```
USO 1 — Capa 3 (análisis estratégico del grid)

USO 2 — LiquidityAnalyzer (nuevo en v5)
  Frecuencia: al arrancar + cada 2 horas

  Tiempo disponible: < 2 segundos ideal
```

### Capa 3 — Usar Sonnet 4.6 (mantener lo actual)

```typescript
// En lib/analysis/layer3-agent.ts
// Mantener:
model: 'claude-sonnet-4-6'
```

La Capa 3 recibe contexto complejo con señales contradictorias
y necesita razonar sobre ellas simultáneamente. Sonnet 4.6
maneja esto mejor que Haiku porque:

```
Haiku en Capa 3:
  ✅ Responde el JSON correcto
  ❌ Razonamiento más superficial
  ❌ Tiende a responder "keep" en situaciones ambiguas
     cuando debería decir "shift_down"
  ❌ Menos útil el campo "reasoning" para debuggear

Sonnet 4.6 en Capa 3:
  ✅ Razona bien sobre señales contradictorias
  ✅ El campo "reasoning" es genuinamente útil
  ✅ Detecta patrones inusuales de mercado
  ✅ Vale el costo dado que solo corre ~6 veces/día
```

---

### LiquidityAnalyzer — Sin modelo de IA

```
El LiquidityAnalyzer NO usa ningún modelo de lenguaje.
Es matemática pura y determinista:

  OBI ratio   = bids_volume / asks_volume  → número
  CVD delta   = buy_volume - sell_volume   → número
  FundingRate = tasa actual                → número
  
  Combinar con pesos → GridBias → decisión

No hay ambigüedad lingüística ni razonamiento complejo.
Usar un LLM aquí sería como usar IA para sumar 2 + 2.
El algoritmo de combinación definido en este documento
es suficiente y más confiable que un modelo.
```

### Texto legible del GridBias — Usar Haiku (opcional)

En lugar de mostrar números crudos en el dashboard,
se puede usar Haiku para generar el campo `summary`
del GridBias en lenguaje natural:

```typescript
// En lib/analysis/liquidityAnalyzer.ts
// Solo para generar el texto descriptivo del bias:

async function buildSummary(bias: GridBias): Promise<string> {
  const response = await anthropic.messages.create({
    model: 'claude-haiku-4-5-20251001',  // Haiku — rápido y barato
    max_tokens: 100,
    messages: [{
      role: 'user',
      content: `En máximo 2 oraciones, explica este sesgo de mercado
                para un trader. Sin tecnicismos innecesarios.
                
                OBI: ${bias.signals.obi.ratio}x (${bias.signals.obi.bias})
                CVD: ${bias.signals.cvd.delta > 0 ? '+' : ''}${bias.signals.cvd.delta} XRP (${bias.signals.cvd.trend})
                Funding: ${bias.signals.fundingRate.rate}%
                Sesgo: ${bias.direction} ${bias.strength}/100
                Grid: ${bias.levelsAbove} niveles arriba, ${bias.levelsBelow} abajo`
    }]
  })
  return response.content[0].text
}

// Resultado en dashboard:
// "Hay 2x más dinero queriendo comprar que vender. El dinero
//  institucional lleva 4 horas acumulando XRP — grid sesgado
//  hacia arriba con 8 niveles de venta vs 4 de compra."
```

Este uso de Haiku es **completamente opcional**. Si no se implementa,
el `summary` se genera con una función de texto determinista
(sin IA) usando plantillas predefinidas:

```typescript
// Alternativa sin IA — función de texto determinista:
function buildSummaryStatic(bias: GridBias): string {
  const dir = bias.direction === 'bullish' ? 'alcista' : 
              bias.direction === 'bearish' ? 'bajista' : 'neutral'
  const strength = bias.strength > 60 ? 'fuerte' :
                   bias.strength > 30 ? 'moderado' : 'débil'
  return `Sesgo ${dir} ${strength} (${bias.strength}/100). ` +
         `Grid: ${bias.levelsAbove} niveles arriba, ${bias.levelsBelow} abajo.`
}
```
