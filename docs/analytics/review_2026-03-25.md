## Las 4 causas raíz — en orden de impacto

### 1. `fetchClosedOrders` en testnet siempre daba timeout

El bot detecta fills así:
```
ciclo → fetchClosedOrders() → ¿alguna orden mía se cerró? → colocar orden contraria → trade contado
```

En las sesiones reales, `fetchClosedOrders` llamaba el endpoint `allOrders` de testnet
sin límite de tiempo ni ventana de fecha — eso devuelve todo el historial desde el inicio
de la cuenta testnet. Con historial largo, Binance testnet tarda >10s y la llamada expira.
El bot nunca supo que una orden se había llenado. Puede que Binance sí ejecutara alguna
compra, pero el bot no lo vio.

La sesión 23:48 tiene 1 trade — probablemente en ese ciclo el timeout no ocurrió, detectó
1 fill, intentó colocar la sell contraria, pero falló por la causa #2.

### 2. Sin XRP disponible para las órdenes sell

Con `ACTIVE_PERCENT=20%` y 1,957 XRP en testnet:
- XRP activo = 1,957 × 20% = 391 XRP
- `amountPerLevel` calculado desde USDT → ~509 XRP por nivel
- El bot intentaba colocar sells de 509 XRP pero solo tenía 391 totales

Binance rechazó las sells con "insufficient balance". El trade del 23:48 probablemente fue:
1 buy detectado → sell rechazada por balance insuficiente → 0 ganancias registradas.

### 3. Grid completamente demasiado estrecho para la volatilidad del día

Durante la tarde/noche la volatilidad fue 45.5% en 24h. Los rangos del grid fueron:

| Sesión | Rango grid | % de volatilidad cubierta | Resultado esperado |
|---|---|---|---|
| 22:14–22:24 | 6% (conservative) | 13% de 45.5% | Precio escapa en minutos |
| 22:42 | 10% (aggressive) | 22% de 45.5% | Sigue siendo insuficiente |
| 23:14 | 6% (conservative) | 13% de 45.5% | Peor que antes |
| 23:33–23:48 | 12% (aggressive manual) | 26% de 45.5% | Aún 1/4 del rango real |

Ninguna sesión nocturna tuvo un rango suficiente. Con 45% de volatilidad el precio se movía
3–6% en minutos, el grid quedaba abandonado casi instantáneamente.

### 4. Sesiones demasiado cortas — thrashing de estrategia

Las órdenes límite necesitan tiempo para llenarse. El patrón del día fue:

```
Arrancar → ver 0 trades → parar → cambiar config → arrancar → ver 0 trades → ...
```

Cada restart cancela todas las órdenes activas en Binance y las re-coloca. Las sesiones de
1–4 minutos nunca dieron tiempo a que el mercado llegara al precio de las órdenes.
La sesión de 15m (23:48) fue la única que duró lo suficiente para ver 1 trade.

El tiempo mínimo recomendado antes de evaluar si el bot está funcionando: **30–60 minutos.**

## Por qué MOCK funcionó y REAL no

| Aspecto | MOCK | REAL |
|---|---|---|
| Detección de fills | Simulada aleatoriamente (25% por ciclo) | Dependía de `fetchClosedOrders` → timeout |
| Balance | Ficticio, ilimitado | XRP insuficiente para sells |
| `SIZING_BASE_AMOUNT` | 0.0013 XRP (microscópico) | 1 XRP (correcto pero choca con balance activo) |
| Timeouts | Ninguno | >10s por ciclo → 0 fills detectados |

## Correcciones aplicadas el 2026-03-26

| Problema | Fix |
|---|---|
| `fetchClosedOrders` timeout | `since = Date.now() - 2h` + `timeout: 30000` + `recvWindow: 60000` |
| Grid estrecho para volatilidad extrema | Rango dinámico = `averageDailyRange × 1.2`, máximo 30% |
| Sells rechazadas por balance | Ajuste automático: `freeBase × activePercent / sellLevelCount` |
| Config incorrecta para vol >15% | Conservative → Aggressive con rango dinámico |
| Config selector sin contexto de rango | `averageDailyRange` ahora se pasa a `selectOptimalConfig` |
| Layer 3 sin datos de rango típico | `averageDailyRange` y cobertura del grid añadidos al contexto de Claude |
